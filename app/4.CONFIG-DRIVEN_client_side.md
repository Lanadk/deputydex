# Config Driven UI

## C'est quoi ?

Le **Config Driven UI** est un pattern d'architecture frontend où le **layout et le contenu d'une page sont déclarés dans une configuration**, séparément du code de rendu.

Au lieu d'écrire du JSX directement dans chaque page, on déclare **ce qui doit s'afficher** dans un fichier de config, et un moteur de rendu générique s'occupe du reste.

```
Config (quoi afficher)  →  Renderer (comment l'afficher)  →  UI
```

---

## Le problème qu'il résout

### Sans Config Driven — approche naïve

```tsx
// page-stats.tsx
export default function StatistiquesPage() {
    return (
        <div>
            <BarChart title="Répartition par âge" data={ageData} />
            <DonutChart title="Genre" data={genreData} />
            <BarChart title="Professions" data={professionData} />
            {/* ... 20 autres charts hardcodés ... */}
        </div>
    );
}
```

**Problèmes :**
- Ajouter un chart = modifier la page
- Réutiliser un chart sur une autre page = copier-coller
- Changer l'ordre = refactorer du JSX
- 10 pages × 20 composants = 200 imports à gérer

---

## La solution — Config Driven

On sépare en trois couches :

### 1. Le Registry — *ce qui existe*

```ts
// stats.registry.ts
export const STATS_REGISTRY: StatConfig[] = [
    {
        id: "age-distribution",
        title: "Répartition par tranche d'âge",
        theme: "demographie",
        gatewayFn: async (legislature) => ({ type: "bar", data: [...] }),
    },
    {
        id: "genre",
        title: "Répartition hommes / femmes",
        theme: "demographie",
        gatewayFn: async (legislature) => ({ type: "donut", data: [...] }),
    },
];
```

> Le registry est une **bibliothèque**. Il ne sait pas où ni comment ses entrées seront affichées.

---

### 2. La Config de page — *ce qui s'affiche et comment*

```ts
// sections.config.ts
export const STATISTICS_SECTIONS = [
    {
        id: "demographie",
        label: "Profil démographique",
        icon: Users,
        cols: 2,
        blocks: [
            {
                type: "paragraph",
                colSpan: 2,
                items: [
                    { type: "highlight", content: "L'Assemblée nationale..." },
                    { type: "kpi", label: "Âge moyen", value: "49,3 ans", trend: "down" },
                ],
            },
            { type: "stat", config: stat("age-distribution"), colSpan: 2 },
            { type: "stat", config: stat("genre") },
            { type: "table", ...table("professions-deputes"), colSpan: 2 },
        ],
    },
];
```

> La config déclare **quoi**, **où** et dans quel **ordre**. Pas de JSX, pas de logique.

---

### 3. Le Renderer — *le moteur générique*

```tsx
// BlockSectionRenderer.tsx
export const BlockSectionRenderer = ({ block, legislature }) => {
    switch (block.type) {
        case "stat":      return <StatCard config={block.config} legislature={legislature} />;
        case "paragraph": return <BlockParagraphRenderer items={block.items} />;
        case "table":     return <BlockTableRenderer config={block} legislature={legislature} />;
    }
};
```

> Le renderer ne sait pas **ce qu'il affiche** — il sait juste **comment afficher chaque type**.

---

### 4. La page — *juste une boucle*

```tsx
// page.tsx
export default function StatistiquesPage() {
    const { legislature } = useLegislature();

    return (
        <PageContentLib>
            {STATISTICS_SECTIONS.map((section) => (
                <AnchorSectionBlockLib key={section.id} {...section}>
                    {section.blocks.map((block, i) => (
                        <BlockSectionRenderer
                            key={i}
                            block={block}
                            legislature={legislature?.number ?? 17}
                        />
                    ))}
                </AnchorSectionBlockLib>
            ))}
        </PageContentLib>
    );
}
```

> La page est **vide de logique** — elle délègue tout à la config et au renderer.

---

## Les types de blocks disponibles

| Type          | Contenu                                              | Composant rendu                                        |
|---------------|------------------------------------------------------|--------------------------------------------------------|
| `"chart"`     | `ChartConfig` (chart)                                | `BlockChartRenderer` → `BarChartLib`, `DonutChartLib`… |
| `"paragraph"` | `ParagraphItem[]` (text, highlight, list, kpi)       | `BlockParagraphRenderer` ->`<p/>`                      |
| `"table"`     | `TableConfig` (colonnes, filtre, pagination, export) | `BlockTableRenderer` ->`TableLib`, ->`TableActionsLib` |

---

## Ajouter un nouveau chart

**1. Ajouter au registry :**
```ts
// stats.registry.ts
{
    id: "mon-nouveau-chart",
    title: "Mon chart",
    theme: "demographie",
    gatewayFn: async (legislature) => ({ type: "bar", data: [...] }),
}
```

**2. Référencer dans la config :**
```ts
// sections.config.ts
{ type: "stat", config: stat("mon-nouveau-chart"), colSpan: 2 }
```

**C'est tout.** Zéro modification de la page ou du renderer.

---

## Ajouter un nouveau type de block

Si demain on veut un block `"map"` ou `"timeline"` :

**1. Étendre le type :**
```ts
// block-section-renderer.tsx
export type SectionBlock =
    | { type: "stat";      ... }
    | { type: "paragraph"; ... }
    | { type: "table";     ... }
    | { type: "map";       ... }  // ← nouveau
```

**2. Ajouter le cas dans le renderer :**
```tsx
case "map": return <BlockMapRenderer config={block} />;
```

**3. L'utiliser dans n'importe quelle config de page.**

---

## Ce que ce n'est PAS

| Pattern | Différence |
|---------|------------|
| **Data Driven** | La config vient d'une API/BDD au runtime — modifiable sans redéploiement |
| **Hardcodé** | JSX direct dans la page — pas de séparation config/rendu |
| **Config Driven** ✓ | Config statique dans le code — typée, versionnée, scalable |

---

## Avantages

- **Scalable** — 50 pages × 30 blocks sans duplication de code
- **Typé** — TypeScript garantit que chaque block est valide à la compilation
- **Séparation des responsabilités** — données / layout / rendu sont indépendants
- **Onboarding facile** — un dev peut ajouter une section sans comprendre le renderer
- **Testable** — les `gatewayFn` sont des fonctions pures, testables unitairement

## Inconvénients

- **Overhead initial** — plus complexe à mettre en place qu'un JSX direct
- **Debugging indirect** — une erreur dans la config peut être moins lisible qu'une erreur JSX
- **Rigidité des types** — ajouter un nouveau type de block demande de modifier le type union

## Conclusion
La paresse bien placée c'est souvent le meilleur moteur de conception.
