# 📚 Result Pattern dans une architecture Clean

## 🎯 Pourquoi ?

En JavaScript, le `throw` est coûteux — il capture une stack trace à chaque fois. Sur des cas métier attendus (un acteur non trouvé, une législature absente), utiliser `throw` c'est traiter un cas normal comme une erreur technique. C'est faux sémantiquement et inutilement coûteux en mémoire.

Le Result pattern force à gérer explicitement les cas d'erreur **sans exception**.

---

## Les types

```ts
export type Success<T> = { success: true; data: T }
export type Failure<E> = { success: false; error: E }
export type Result<T, E = Error> = Success<T> | Failure<E>
```

Un `Result` est soit un succès avec une donnée, soit un échec avec une erreur typée.

---

## Les helpers disponibles

### Créer un résultat

```ts
ok(data)   // crée un Success
err(error) // crée un Failure
```

### Lire un résultat

```ts
isOk(result)   // narrowing TypeScript → accès à result.data
isErr(result)  // narrowing TypeScript → accès à result.error
```

### Extraire la valeur

```ts
// Valeur par défaut si erreur
unwrapOr(result, defaultValue)

// Fonction de fallback si erreur
unwrapOrElse(result, (error) => fallback)

// ⚠️ Throw si erreur — à utiliser avec parcimonie
// Réservé aux cas où l'échec est véritablement inattendu
unwrap(result)
```

## Les erreurs génériques

Plutôt que des strings littérales isolées par use case, on définit un type d'erreurs partagé :

```ts
// _shared/errors/app-errors.ts

export type AppError =
  | { code: "NOT_FOUND"; message?: string }
  | { code: "FORBIDDEN"; message?: string }
  | { code: "DB_ERROR"; message?: string }
  | { code: "VALIDATION_ERROR"; message?: string }
```

Le use case retourne alors :

```ts
export async function getCurrentLegislatureUseCase(
  repository: ILegislaturesRepository
): Promise<Result<LegislatureDTO, AppError>> {
  const entity = await repository.getCurrent();

  if (!entity) return err({ code: "NOT_FOUND" });

  return ok(mapLegislatureToDTO(entity));
}
```

La route lit le code pour décider du statut HTTP :

```ts
if (isErr(result)) {
  const status = result.error.code === "NOT_FOUND" ? 404 : 500;
  return NextResponse.json({ error: result.error.code }, { status });
}

return NextResponse.json(result.data);
```

## Wrapper les erreurs techniques en erreurs génériques

Sans ce pattern, une erreur Prisma traverse toutes les couches et se retrouve dans la route API — l'infrastructure fuit dans le domaine.

L'idée : **le repository catch les erreurs techniques et les traduit en `AppError`**.

```ts
// infrastructure/legislatures/repositories/prisma-legislatures.repository.ts

async getCurrent(): Promise<Result<LegislatureEntity, AppError>> {
  try {
    const row = await prisma.paramLegislature.findFirst({ ... });
    if (!row) return err({ code: "NOT_FOUND" });
    return ok(row);
  } catch (e) {
    console.error(e); // log technique ici, pas dans le use case
    return err({ code: "DB_ERROR", message: "Failed to fetch current legislature" });
  }
}
```

Le use case ne voit jamais Prisma — il reçoit soit une entité, soit un `AppError` propre :

```ts
export async function getCurrentLegislatureUseCase(
  repository: ILegislaturesRepository
): Promise<Result<LegislatureDTO, AppError>> {
  const result = await repository.getCurrent();

  if (isErr(result)) return result; // on propage sans transformer

  return ok(mapLegislatureToDTO(result.data));
}
```

### Pourquoi c'est important

Chaque couche parle son propre langage :

```
DB (PrismaClientKnownRequestError)
  ↓ wrappé par le repository
infrastructure  →  { code: "DB_ERROR" }
  ↓ propagé par le use case
domain          →  AppError
  ↓ traduit par la route
HTTP            →  404 / 500
```

Sans ce wrapping, une `PrismaClientKnownRequestError` peut remonter jusqu'à la route et exposer des détails d'infrastructure. Avec ce wrapping, chaque couche reste hermétique — une erreur DB ne traverse jamais le domaine sous sa forme brute.

---

## Où utiliser quoi

| Situation | Mécanisme |
|---|---|
| Cas métier attendu (not found, forbidden...) | `Result` + `err({ code: "..." })` |
| Erreur technique dans le repository | `try/catch` → `err({ code: "DB_ERROR" })` |
| Erreur technique vraiment inattendue | `try/catch` dans la route API |
| Erreur côté front gateway | `throw new Error(...)` |
| 404 côté front gateway | `return null` |
| Cas sans erreur métier possible | `Result<T, AppError>` — liste vide = succès |

## Référence rapide

```ts
// Créer
return ok(data);
return err({ code: "NOT_FOUND" });

// Lire — early return sur l'erreur
if (isErr(result)) {
  // gérer l'erreur
}
return result.data; // TypeScript sait que c'est un Success ici (pas besoin d'unwrap car nawrowing)

// Valeur par défaut
const data = unwrapOr(result, []);

// Fallback
const data = unwrapOrElse(result, (error) => handleError(error));

// ⚠️ Throw — cas vraiment exceptionnel uniquement ... meme pas sur que ça existe dans notre codebase pour l'instant
const data = unwrap(result);
```
