# Stratégie API & Cache — Tableaux

## Contrat API

L'API renvoie dans une seule réponse les données **et** les métadonnées de filtrage :

```ts
{
    rows: ProfessionRow[],
    filterFields: [
        { field: "nom",     label: "Nom",     type: "string" },
        { field: "famille", label: "Famille", type: "enum",  enumValues: ["Juristes", "Enseignants"] },
        { field: "groupe",  label: "Groupe",  type: "enum",  enumValues: ["EPR", "SOC", "RN"] },
    ],
    sortOptions: [
        { id: "nom-asc", label: "Nom A→Z", field: "nom", direction: "asc" },
    ]
}
```

Le backend calcule les `enumValues` dynamiquement selon la législature :

```sql
SELECT DISTINCT groupe FROM deputes WHERE legislature = ?
```

Le frontend ne hardcode rien — il reçoit, il affiche.

---

## Pourquoi une seule requête

Deux requêtes séparées doublent la latence sans bénéfice :

```ts
// ❌ Deux allers-retours réseau
const rows    = await fetch("/api/professions");
const filters = await fetch("/api/professions/filters");

// ✓ Un seul aller-retour
const { rows, filterFields, sortOptions } = await fetch("/api/professions");
```

Le surcoût en payload est négligeable — les métadonnées représentent quelques dizaines d'octets.

---

## gatewayFn

La signature de `gatewayFn` dans `TableConfig` reflète ce contrat :

```ts
gatewayFn: (legislature: number) => Promise<{
    rows: TRow[];
    filterFields?: FilterField[];
    sortOptions?: SortOption[];
}>
```

La config statique dans le registry sert de **fallback pour le dev/mock**. En production, l'API est la source de vérité.

`BlockTableRenderer` merge les deux sources :

```ts
const { rows, filterFields, sortOptions } = await config.gatewayFn(legislature);
// filterFields de l'API écrase la config statique si présent
```

## Stratégie de cache

Les données parlementaires sont mises à jour **une fois par semaine**. Le cache est donc configuré avec une revalidation hebdomadaire :

```ts
const response = await fetch("/api/professions", {
    next: { revalidate: 604800 } // 7 jours
});
```

`rows`, `filterFields` et `sortOptions` sont cachés ensemble — un seul hit API par semaine par endpoint.

---

## Invalidation manuelle après update

Plutôt qu'un revalidate temporel fixe, l'invalidation est déclenchée après le job d'import hebdomadaire :

```ts
// Tagging des requêtes
fetch("/api/professions", {
    next: { tags: ["statistics"] }
});

// Après le job d'import
revalidateTag("statistics");
```

Le cache est invalidé **exactement** quand de nouvelles données arrivent, pas avant. Zéro requête inutile dans la semaine.
